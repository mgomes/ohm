# ADR 0009: Replayable Requests

## Status

Accepted

## Context

Ohm should make web application debugging and regression testing unusually
strong. One promising feature is the ability to capture enough information about
a request to reproduce it later as a local debug run or generated test.

This should not mean recording live user sessions, continuously storing all
production traffic, or serializing Go's `context.Context`. Live session
recording would be expensive, privacy-sensitive, and operationally risky.
`context.Context` is for cancellation, deadlines, and scoped values, not durable
execution records.

The useful feature is replayable requests: explicit, scrubbed request snapshots
that can be replayed through the application's handler stack under test.

## Decision

Ohm will treat replayable requests as an early framework capability, not a
distant post-1.0 feature.

Replayable requests should be explicit and test-oriented. The first version
should focus on local development, integration tests, and bug reproduction. It
should not include automatic production traffic capture.

A replay snapshot may include:

- HTTP method.
- Path.
- Query string.
- Route pattern and route params when available.
- Selected request headers.
- Request body, behind explicit opt-in and size limits.
- Authenticated principal reference, not raw cookies or session secrets.
- Request id.
- Application version when available.
- Environment name.
- Feature flag values when available.
- Timestamp metadata.
- Scrubbed params and framework metadata.

Replay snapshots must use the same scrubbing policy as framework logging and
error reporting. Sensitive values should be redacted before snapshots are
written.

The initial replay flow should look like:

```text
myapp replay ./tmp/replays/login-failure.json
ohm generate test-from-replay ./tmp/replays/login-failure.json
```

Replay should run through the application's real handler stack using Go test
infrastructure such as `httptest`.

The framework should make deterministic boundaries possible over time:

- Clock.
- Random id generation.
- External HTTP clients.
- Email delivery.
- File writes.
- Database state.
- Feature flags.

The first version does not need to solve all deterministic boundaries, but it
must avoid pretending replay is perfectly deterministic when the application
uses uncontrolled time, randomness, external services, or database state.

## Consequences

Replayable requests can become a defining developer-experience feature for Ohm.
They support closed-loop debugging by turning a failing request into something
that can be rerun locally and converted into a regression test.

Keeping the feature explicit avoids the performance, privacy, and storage costs
of live session recording.

The feature depends on earlier framework decisions: structured scrubbing,
application CLI commands, route introspection, test helpers, and eventually
stable framework-level request abstractions.

Ohm will need clear snapshot formats and compatibility rules once replay files
are generated by real applications.

## Non-Goals

- Serialize Go's `context.Context`.
- Record every production request by default.
- Store raw cookies, authorization headers, passwords, tokens, or session
  secrets.
- Guarantee deterministic replay without explicit deterministic application
  boundaries.
- Build a browser session recorder.

## Open Questions

- Should replay snapshots be JSON, another structured format, or versioned Go
  fixtures?
- Where should generated applications store local replay files by default?
- Should replay support database fixture references in the first version?
- Should replay support recorded outbound HTTP responses in the first version?
- How should authenticated principals be represented without coupling replay to
  one auth implementation?
